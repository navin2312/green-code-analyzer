The Problem

Software is a hidden energy consumer

Code that works correctly can still waste enormous amounts of energy.
These inefficiencies are invisible during normal code review.

🔁 Unnecessary CPU Wake-ups

Polling APIs every 200ms wakes the CPU 5× per second — even when nothing has changed. Event-driven alternatives use zero idle energy.

📡 N+1 Network Requests

Fetching 100 items one by one in a loop sends 100 HTTP requests. One batched call does the same work with 1% of the network energy.

🔄 Busy-Wait Loops

A Python while True: without sleep burns 100% of a CPU core continuously, preventing it from ever entering a low-power state.

🐼 Inefficient Data Processing

pandas.iterrows() is up to 1000× slower than vectorised operations — and proportionally more energy-intensive for large datasets.

→ At scale, across millions of users and thousands of servers, these patterns translate into measurable CO₂ emissions and electricity costs.

Our Solution

Automated energy review — right in the PR

Green Code Analyzer integrates into the GitHub pull request workflow. Developers get energy feedback the same way they get code review comments — without changing how they work.

🤖 Automatic

Triggers on every pull request. No manual steps. No configuration required beyond adding one workflow file.

🎯 Precise

Only analyses lines the developer actually changed in this PR — not the entire codebase. Zero noise from existing code.

💡 Actionable

Every finding includes an exact line number, an explanation of why it wastes energy, and a corrected code example.

📊 Graded

PRs receive an energy efficiency grade from A+ to D based on the severity and count of issues found.

🌍 Quantified

Estimates the Wh/day and grams of CO₂ saved if the issues are fixed — making the impact tangible.

🖥️ CLI Too

Developers can run green-code check locally before pushing — catching issues even earlier.

diff-parser.js

Parses the unified diff format. Extracts only added lines (+) per file, keeping context lines for loop-detection lookback.

analyzer.js

Runs each pattern's detect(lines, index) function. Only reports findings on added lines — context is read-only.

energy-estimator.js

Sums energy points per finding. Maps to A+–D grade. Calculates rough Wh/day and CO₂ savings.

reporter.js

Generates markdown for PR comments (with shields.io badge), ANSI terminal output for CLI, and JSON for integrations.

Scoring System

Energy efficiency grading — A+ to D

Each pattern has an energy weight (5–35 points based on severity). Findings are summed into a total score that maps to a letter grade.

A+
Score: 0
Excellent — no issues found
A
Score: 1–15
Good — minor concerns
B
Score: 16–40
Fair — some improvements needed
C
Score: 41–80
Needs Work — significant issues
D
Score: 81+
Critical — fix before merging

🔴 Critical = 35 pts

e.g. fetch() in loop — one occurrence alone pushes to grade C

🟠 High = 20 pts

e.g. setInterval < 1s — two occurrences reach grade B

🔵 Low = 5–6 pts

e.g. console.log in loop — many needed to affect the grade

GitHub Action

Zero-friction PR integration

Add one workflow file to any repository — the action handles everything else automatically.

# .github/workflows/energy-check.yml
name: Energy Check
on:
  pull_request:
permissions:
  pull-requests: write
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: navin2312/green-code-analyzer@main
        with:
          github-token: ${{ secrets.GITHUB_TOKEN }}
          fail-on-issues: 'false'
          severity-threshold: 'low'

The action outputs three values usable in downstream steps:

outputs.grade

Letter grade: A+, A, B, C, or D

outputs.energy-score

Total numeric score (sum of all finding weights)

outputs.findings-count

Number of issues detected above the severity threshold

Set fail-on-issues: 'true' to enforce a green gate — PRs with energy issues will fail the check.

CLI Tool

Catch issues before you push

Install globally and run against any file, your current git diff, or staged changes.

# Install
npm install -g green-code-analyzer

# Check specific files
green-code check src/app.js utils.py

# Check uncommitted git changes
green-code check --git-diff

# Check only staged changes
green-code check --staged

# Output as JSON (for CI integrations)
green-code check src/app.js --format json

# Fail with exit code 1 if issues found
green-code check src/app.js --fail-on-issues

# List all known patterns
green-code list-patterns
